# Character dossiers

## Purpose

Files in `canon/characters/` define durable character logic for the correct campaign phase. They are not current-state logs and they are not dialogue phrase banks.

Use them with:

- `canon/characters/CHARACTER_DOSSIER_APPLICATION.md`;
- relevant event tiles in `continuity/events/`;
- `continuity/EVENT_INDEX.md`;
- `continuity/KNOWLEDGE_LEDGER.md`;
- `canon/relationships/RELATIONSHIP_STATE.md`;
- current character-state files when they exist;
- `canon/prose/CHAPTER_PROSE_AND_CHARACTER_VOICE_PASS.md`;
- `templates/CHARACTER_DOSSIER_APPLICATION_TEMPLATE.md`.

## Stable versus current information

A dossier controls:

- campaign-stage presentation;
- spoiler rail;
- physical habits;
- voice and pressure behavior;
- practical competence;
- story position;
- response logic;
- Mielle contact boundaries;
- setting ties.

Approved MiE events and current beliefs are controlled by continuity files. The `## Event index` section of each dossier points to `continuity/EVENT_INDEX.md`; do not maintain duplicate event lists inside stable dossiers.

## Application rule

Reading a dossier or writing one plausible line does not prove application. Every decision-relevant character must leave a productive or restrictive consequence in the scene and pass removal, replacement, knowledge, phase and existing-ties checks under `canon/characters/CHARACTER_DOSSIER_APPLICATION.md`.

Background presence is allowed. Do not force every present character to speak, react or perform a dossier trait.
